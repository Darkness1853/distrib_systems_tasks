# Вопросы для самопроверки

1. Зачем нужен файл `.proto`? Почему мы не пишем классы сообщений сразу на Python?

Файл `.proto` нужен потому что:
- Это язык описания интерфейсов, он более эффективный для генерации на разных языках
- Если изменить тип поляв `.proto`, код не скомпилируется. В gRPC контракт — это закон.
- `.proto` файл генерирует сам. Если вручную писать код - это может быть неэффективно.

2. Что такое Unary RPC? Чем он концептуально похож на REST запрос?

Unary RPC — это самый простой тип взаимодействия в gRPC.
- Также как и в REST, клиент делает один запрос и получает один ответ
- Оба подхода следуют модели "запрос-ответ"
- Оба напоминают обычный вызов функции, но по сети

В REST данные передаются текстом (JSON), а в gRPC — бинарными данными (Protocol Buffers).

3. Почему gRPC обычно эффективнее REST (JSON) по процессору и трафику?

- Скорость сериализации (JSON нужно парсить текст, что медленно для машины, но понятней для человека. А Protobuf работает с бинарными данными, что значительно быстрее)
- Меньше трафика (JSON передает данные в текстовом виде.
Protobuf передает только сжатые бинарные данные)

4. Что такое "обратная совместимость" в Protobuf и зачем нужны номера полей (тэги, например `= 1`)?

Номера полей позволяют Protobuf понимать, какое поле к какому значению относится, даже если порядок полей изменился.

"обратная совместимость":
- Можно добавлять новые поля ведь они не повлияют на старых клиентов
- Нельзя менять теги существующих полей
- Нельзя менять тип существующих полей

5. Что такое Stub (стаб) на клиенте и Servicer на сервере?

Stub - заглушка

На сервере:
- Класс, который нужно реализовать на сервере, обрабатывает входящие запросы
- Вместе Stub и Servicer образуют полный контракт между клиентом и сервером

На клиенте:
- Сгенерированный код на стороне клиента, который выглядит как обычный объект
- После вызова он сам упаковывает данные и отправляет в сеть

6. В каких случаях вы бы выбрали REST, а в каких — gRPC?

gRPC:
- Если нужна максимальная производительность
- Меньше места занимает
- Поддержка/генерация разных языков
- Нужна строгая типизация

REST:
- API публичное
- Простая отладка, ведь JSON легче читается 
- Без генерации кода, вручную
