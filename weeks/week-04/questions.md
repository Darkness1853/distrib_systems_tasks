# Вопросы для самопроверки

1. Почему в микросервисах нельзя просто использовать `BEGIN TRANSACTION ... COMMIT` для операций, затрагивающих несколько сервисов?

В микросервисах у каждого сервиса своя база данных. ACID-транзакции работают только в пределах одной БД. Нельзя сделать COMMIT в базы разных сервисов одновременно — это нарушит согласованность.

2. В чем разница между Оркестрацией и Хореографией? Какой подход проще для старта, а какой лучше масштабируется?

Оркестрация: центральный координатор управляет шагами (проще для старта)

Хореография: сервисы общаются через события (лучше масштабируется)

3. Что такое "компенсирующая транзакция"? Приведите пример.

Это отмена предыдущего действия. Пример: если оплата не прошла, система автоматически отменяет резерв билета (CANCELLED).

4. Может ли компенсирующая транзакция завершиться ошибкой? Что делать в таком случае?

Да, может. Нужен retry-механизм (повторные попытки) пока компенсация не выполнится.

5. Что такое "согласованность в конечном счете" (Eventual Consistency)?

Данные станут согласованными не сразу, но в конечном счете. Например: после оплаты билет не сразу меняет статус, но через несколько секунд станет PAID.
