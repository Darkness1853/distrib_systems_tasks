# Вопросы для самопроверки

1. Зачем нужны `variables` в GraphQL запросе? Почему нельзя просто склеивать строки?

- Код более понятней и структурированней
- Защищает от инъекций
- Проблема с валидацией

2. Из каких полей состоит стандартный JSON-запрос к GraphQL серверу?

```
  {
    "query": "query($id: ID!) { user(id: $id) { name } }",
    "variables": { "id": 1 }
  }
```

3. Почему у GraphQL обычно только один URL endpoint?

- Потому что клиент определяет какие данные ему нужны. За один запрос все необходимые данные
- Это уменьшает количество запросов, которые нужно отправлять серверу


4. Что означает наличие поля `errors` в ответе сервера? Может ли оно прийти вместе с `data`?

Наличие поля `errors` означает, что произошла ошибка при выполнении запроса
`data` может прийти вместе с ошибкой. Например, когда мы запросили два поля, один из них был выполнен успешно, другой с ошибкой

5. Как клиент должен обрабатывать ошибки, если HTTP статус ответа 200?

Нужно чтоб клиент проверял наличие поля `errors` в ответе. И Обрабатывать такие случаи соответсвенно

6. В чем плюсы использования типизированных клиентов (например, Apollo Client или Relay) перед простыми HTTP-запросами?

- не нужно писать HTTP-запросы вручную
- удобная работа с данными. Можно запрасывать необходимые данные
- Легче разрабатывать 
- Ускоряет загрузку повторных запросов и уменьшает количество запросов к серверу
