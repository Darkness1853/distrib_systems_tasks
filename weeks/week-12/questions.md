# Вопросы для самопроверки

1. Почему мы создаем Deployment, а не сразу Pod?

Потому что **Pod** - не сможет восстановиться при завершении работы, то есть Pod не пересоздается автоматически. 

**Deployment** следит за их количеством, если один из **Pod** завершает работу,  Deployment сразу же создает новый.

2. В чем разница между Liveness и Readiness пробами? Что произойдет, если провалится каждая из них?

**Liveness probe** - определяет находится ли контейнер в живом состоянии. Liveness  перезапускает контейнер если контейнер мертв

**Readiness probe** - определяет готов ли контейнер принимать входящий трафик. Если нет, то контейнер не перезапускается, а временно исключается из списка. Не получает запросы до тех пор, пока  проверка не будет пройдена.

3. Что такое "requests" и "limits" для CPU и памяти? Почему важно их указывать?

**Requests** определяет минимальное кол. ресурсов, которое гарант. резерв. для контейнера на узле.

**Limits** устанавливает макс. кол. ресурсов, которое контейнер может использовать.

- Если указывать **Requests** то  Kubernetes  использует эти значения для определения размещения Pod на подходящем узле.
- Также не позволяет чтоб один контейнер потреблял все ресурсы узла, влияя на работу соседних
- Защищает / стабилизирует узел, при превышении лимита памяти контейнера.

4. Как Service находит свои Pod-ы? (Подсказка: Labels & Selectors).

Service находит свои Pod-ы с помощью  Labels (меток) и Selectors (селекторов).

В **Deployment** каждому Pod присваиваются метки, ```app: users-app```

В **Service** указывается селектро с теми же ключами. При создании и удалении Pod Service обновляет список endpoint.

5. Какой тип сервиса (ClusterIP, NodePort, LoadBalancer) используется для внутреннего общения, а какой - для внешнего доступа?

Тип **ClusterIP** предназначен для внутреннего общения внутри кластера Kubernetes.

Тип **NodePort** и **LoadBalancer** предгазначен для обеспечения доступа к приложению извне.

6. Что такое "Rolling Update" в контексте Деплоймента?

Rolling Update (обновление без простоя) - стратегия обновления приложения, при которой старые версии Pod-ов постепенно заменяются новыми версиями без полного отключения сервиса. 

Deployment выполняет обновление согласно заданным параметрам
