# Вопросы для самопроверки

1. Зачем создавать пользовательскую сеть (bridge network) в Docker Compose, если есть дефолтная?

- Для Изоляции - чтоб разные проекты не видели контейнеры друг друга
- DNS - в дефолтной сети контейнеры общаются по IP, а в пользовательской можно прописывать порт

2. В чем разница между `build` и `image` в описании сервиса?

```
build: ./app — Docker берет Dockerfile из папки app и собирает образ сам

image: postgres:15 — Docker скачивает готовый образ
```

3. Что делает `depends_on`? Гарантирует ли он, что приложение в зависимом контейнере полностью инициализировалось?

depends_on гарантирует только порядок запуска, но НЕ готовность сервиса.

Чтобы ждать готовности, необходимо добавить `condition: service_healthy`.

4. Как работают healthcheck-и и как использовать их в `depends_on`?

Healthcheck - механизм проверки работоспособности приложения внутри контейнера, который выполняет команду с заданной периодичностью.

Использование depends_on позволяет запустить зависимый контейнер только после того, как основной контейнер не станет healthy

5. Как передать переменные окружения в контейнер через Compose (минимум 2 способа)?

1) Способ - environment

```
services:
  app:
    environment:
      - DB_HOST=postgres
      - DB_PASSWORD=secret
```

2) Способ - env_file

```
services:
  app:
    env_file:
      - .env
```

2. Почему Docker Compose редко используют в продакшене (хотя это возможно)?

По этим причинам:

**Нет масштабирования** - работает только на одном сервере

**Нет авто-восстановления** - если контейнер упал, Compose не перенесет его на другой хост

**Сложность обновления**  - Трудно обновлять без простоя сервисов
